# 网页豆包实时语音入口设计

## 目标

在 `web/control_simulator.html` 提供与 APP 一致的语音入口：浏览器采集麦克风音频，交由车端豆包实时语音服务完成识别、对话和车载 TTS。网页不得保存或暴露豆包凭据；车辆动作继续经过既有安全路由。

## 架构

`网页麦克风 → 本机 WebSocket 代理 → 车端语音网关 → doubao_voice_node → 豆包实时对话 API`

- 网页采集音频并转换为 16 kHz、单声道 PCM 分片，通过新增 `/ws/voice` 发送。
- 本机 `local_control_ws_proxy.py` 只监听环回地址，将该语音通道转发到小车；现有控制与视频代理不改变。
- 车端语音网关接收二进制 PCM，会话开始/结束事件并转发给 `doubao_voice_node`。
- `doubao_voice_node` 将网页 PCM 送入既有豆包实时会话；识别、回复和状态继续经 `/voice/assistant_result`、`/voice/status` 发布，TTS 仅输出到车载音箱。
- `voice_command_router_node`、`task_manager` 与安全仲裁保持为唯一的任务/运动执行路径。

## 网页交互

- 在任务输入行添加麦克风按钮；单击开始录音，再次单击停止。
- 录音中显示“正在聆听”、录音强调色与实时音量；回传的识别文本、助手回复和错误显示在现有会话区。
- 网页麦克风许可被拒绝、浏览器不支持音频采集、代理/车端连接失败时，均显示可操作的错误提示并恢复可再次录音状态。
- 停止录音代表用户结束本轮；豆包完成识别后自动进入对话及安全任务路由。文本输入保留为调试入口，不构成正式语音路径。

## 验收

1. 页面可申请麦克风权限并通过 `ws://127.0.0.1:8765/ws/voice` 发送 16 kHz 单声道 PCM。
2. 车端可收到音频、豆包返回识别/回复文本，车载音箱播放回复。
3. 查询类语音命令不会产生运动；任务类语音命令仍遵循确认和既有安全链路。
4. 断开、拒绝权限、麦克风不可用及豆包连接失败均不会遗留录音状态或发送运动指令。
